Python 中的函式多載與型別提示： @overload

  • Python
  • Python
  • Overload
  

前言

在 Python 的世界裡，函式多載 (Function Overloading) 並不像 C# 那樣，允許我們定義多個同名但參數簽名不同的函式。然而，在開發複雜的應用程式或函式庫時，我們經常希望同一個函式能根據不同的輸入參數，提供不同的行為或回傳型別。這時候，typing 模組中的 @overload 裝飾器就成了我們的得力助手，它能在不犧牲型別安全性的前提下，為函式提供更豐富、更精確的「使用說明」。

處理問題

問題: 如何在 Python 中為同一個函式提供多種不同的參數簽名 (signature) 和回傳型別提示，以提升程式碼的可讀性、可維護性，並讓型別檢查工具（如 Mypy, Pyright 等）能正確地捕捉潛在錯誤？

解決方法: 使用 @overload 裝飾器來定義函式的多個「型別簽名」，並搭配一個實際的函式實作來處理所有情況。這讓開發者和型別檢查工具都能清楚地知道，在不同呼叫方式下，函式預期接收什麼參數，以及會回傳什麼結果。

處理過程

@overload 的核心概念

@overload 並非實現真正的函式多載，而是為型別檢查工具提供多個函式簽名的「宣告」。你可以將它想像成是為你的函式寫了多份「使用手冊」或「規格說明書」：

  • 每份 @overload 定義都像是一份獨立的「使用手冊」，說明了在特定參數組合下，函式預期接收的參數型別和回傳的結果型別。
  • 這些「使用手冊」只供型別檢查工具參考，它們本身不包含任何實際的程式邏輯。
  • 實際的函式邏輯只會存在於一個沒有 @overload 裝飾的同名函式中。這個函式的簽名（參數和回傳型別）必須能夠兼容所有 @overload 宣告的簽名，確保它能處理所有可能的呼叫情況。

範例分析：ResourceSelector 的 select_resource 函式

讓我們以一個更通用的 ResourceSelector 類別為例，它負責根據資源 ID 和一個旗標來選擇不同類型的資源處理器。

我們希望 select_resource 函式能有以下兩種明確的型別行為：

  1. 如果只提供一個 resource_id，它回傳一個 ResourceBase 類型的處理器。
  2. 如果提供 resource_id 並且明確指定 detailed=True，它回傳一個 DetailedResourceBase 類型的處理器。

程式碼範例

備註：

  • @overload 裝飾器定義的函式體通常只包含 ... (Ellipsis)，表示它沒有實際的執行邏輯，僅用於型別提示。
  • 實際執行的函式必須放在所有 @overload 定義之後，且其簽名必須能夠「覆蓋」所有 @overload 宣告的簽名。
  • 使用 Type[] 是因為我們回傳的是類別本身，而不是類別的實例。

總結

@overload 裝飾器是 Python 型別提示系統中一個強大的工具，它讓你的函式在型別層面擁有「多種面貌」，而底層的實作則統一處理所有情況。這大大提升了 Python 程式碼的型別安全性、可讀性，並改善了開發體驗，尤其是在大型專案或函式庫開發中，能有效減少潛在的型別錯誤。
