前言

在開發 FastAPI 應用程式時，經常會遇到異常處理混亂的問題。特別是在 Repository 層，當業務邏輯異常（如 NotFoundException）被通用的 except Exception 捕獲後，會被錯誤地轉換為系統異常（如 InternalServerException），導致前端無法正確處理 404 錯誤，反而收到 500 內部伺服器錯誤。

本文將介紹如何透過異常分類重構，建立清晰的異常處理架構，讓業務邏輯異常和系統異常能夠被正確區分和處理。

處理問題

原始問題

在 update_user_profile 方法中，當找不到用戶時會拋出 NotFoundException，但這個異常被外層的 except Exception as e: 捕獲，然後轉換成 InternalServerException，導致：

  1. 前端收到 500 錯誤而非預期的 404 錯誤
  2. 異常處理邏輯混亂，難以維護
  3. 每個方法都需要手動添加特定異常處理

解決目標

建立一套清晰的異常分類系統，讓：

  • 業務邏輯異常（404, 400, 401, 403）直接返回給用戶
  • 系統異常自動轉換為 500 錯誤
  • 統一的異常處理模式，易於維護

統一例外處理器的功用與目的

主要功能：

  1. 標準化回應格式：確保所有 API 錯誤回應都有一致的 JSON 格式
  2. 異常分層處理：區分 HTTP 異常和系統異常，給予不同的處理方式
  3. 錯誤日誌記錄：自動記錄未預期的系統錯誤，便於除錯和監控
  4. 安全性保護：避免將系統內部錯誤訊息直接暴露給用戶

使用場景：

  • 當 Repository 拋出 NotFoundException 時，會被 http_exception_handler 處理，返回 404 狀態碼
  • 當發生未預期的系統錯誤時，會被 general_exception_handler 處理，返回通用的 500 錯誤訊息
  • 所有錯誤回應都會有統一的 {"status": "xxx", "message": "xxx"} 格式

架構優點

  1. 語義清晰：明確區分業務邏輯異常和系統異常
  2. 統一處理：所有 Repository 使用相同的異常處理模式
  3. 向後相容：不影響現有的 Service 和 Client 層程式碼
  4. 易於維護：新增異常時只需要繼承適當的基類即可
  5. 錯誤追蹤：系統異常會被自動記錄，便於除錯

備註：

  • 業務邏輯異常（如驗證失敗、資源不存在）應該繼承 BusinessLogicException
  • 系統異常（如資料庫連線失敗、外部服務異常）應該繼承 SystemException
  • 統一例外處理器確保所有錯誤都有一致的回應格式
  • 記得在每個 Repository 方法中都要加入 except BusinessLogicException: raise 的處理
