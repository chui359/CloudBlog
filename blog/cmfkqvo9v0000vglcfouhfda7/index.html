前言

近期在專案中遇到 API 權限錯誤與資料存取錯誤的處理問題，為了確保分工明確，特別釐清 Service 層與 Repository 層的責任界線。本篇將分享思考脈絡、最佳實踐與具體範例，協助讀者建立清晰的錯誤回應設計。

處理問題

主要目標：

  • 明確區分 Service 層和 Repository 層的職責。
  • 針對 API 權限、連線等錯誤，設計適當的錯誤拋回與處理機制。
  • 強化程式碼可維護性與擴充性，避免錯誤訊息被遮蔽或誤判。

處理辦法：

  • Repository 層只負責資料存取與拋出明確 Exception，不包業務邏輯。
  • Service 層負責捕捉錯誤，根據業務需求做適當處理與回應。
  • 透過自訂 Exception 類別，讓錯誤情境更清晰可追蹤。

處理過程

1. 分層責任釐清

  • Repository 層：存取資料來源（如 DB、API），出現權限、連線、資料錯誤時，需拋出明確 Exception，不直接回傳 None 或包裝為業務邏輯。
  • Service 層：負責業務流程與錯誤處理，根據錯誤型態決定如何回應（如回傳 HTTP 狀態碼、錯誤訊息等）。

5. 實作注意事項

  • 不要在 Repository 層包業務邏輯（如判斷權限或回傳業務用錯誤訊息）。
  • 保持 Exception 類別語意明確，便於追蹤與擴充。
  • Service 層可根據錯誤型態，彈性回應前端或進行 log 記錄。

備註：

  • 若有更多層次（如 Handler、Controller），可繼續往上包裝 Exception。
  • 可依據專案規模，調整 Exception 類別細膩度。
