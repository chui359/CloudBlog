FastAPI 專案設計最佳實踐2：Repository Layer優化

前言

在現代後端開發中，分層架構（Layered Architecture）有助於提升系統的維護性、擴充性與可測試性。隨著業務發展，常常會遇到需要切換資料來源（如外部 API、資料庫、快取等）的情境。本文將以具體範例，說明如何以「Repository Layer」進一步優化分層架構，並結合 FastAPI 完成高彈性、低耦合的設計。

問題

前篇 FastAPI 專案設計最佳實踐2：Repository Layer優化 原始專案架構將 Service 層直接綁定資料來源（如 API），雖然分工清楚且易於測試，但當資料來源需要擴充或替換時，Service 層與資料存取邏輯耦合度高，導致維護與擴充困難。如何將業務邏輯與資料存取完全分離，並提升測試與擴充彈性，是本篇文章要解決的核心問題。

解決辦法

1. 定義 Domain Model 與 Pydantic Schema

首先，需釐清「資料結構」及「資料流動方式」。Pydantic Schema 負責資料驗證與序列化，Domain Model 則是各層之間傳遞的資料型態。

說明：
此 schema 定義了專案的基本欄位，方便 API 回傳、資料庫存取及各層資料流動時使用。

2. 建立 Repository 介面與實作

Repository 層的目標是將資料存取抽象化（不論是 API、資料庫還是快取），讓 Service 層不需關心資料來自哪裡。

抽象介面

說明：
抽象介面規範所有資料來源的操作方式，保障 Service 層的程式碼都能「依賴抽象」而非「依賴實作」，達到資料來源的可替換性。

API 實作範例

說明：
這裡的 Repository 實作負責向外部 API 取得、寫入或查詢資料。未來若需改成資料庫存取，只需更換 Repository 實作即可，Service 層邏輯完全不需變動。

假資料/測試實作

說明：
測試時可用這種「假資料實作」來取代真實 API 或資料庫，讓測試環境更純粹、穩定，並可聚焦於業務邏輯驗證。

3. Service 層僅依賴 Repository

Service 層專注於業務邏輯（如流程控制、驗證、資料整合），不負責資料來源細節。

說明：
Service 層只負責「用 Repository 取得資料」並執行業務邏輯。未來若業務流程調整，僅需調整此層，資料來源完全不受影響。

4. API 層依賴注入並統一回應格式

API 層負責「接收外部請求」與「包裝標準回應格式」，同時透過依賴注入（Dependency Injection）靈活切換 Repository 實作。

說明：
API 層負責將外部請求轉成 Service 層可用參數，並將結果包裝成標準回應。透過 Depends，可自由切換 Repository 實作，測試與正式環境皆容易切換。

補充說明：

  • Repository 層可依需求擴充為 API、DB、Cache 多種實作，讓資料取得更彈性。
  • Service 層聚焦於業務邏輯，未來即使有多種資料來源，也無需重構業務流程。
  • 單元測試時只需 Mock Repository，測試更簡潔、易控管。

常見問題

  • Q: 只用 Service 層抽象，為何還需要 Repository 層？
    • A: Service 層抽象主要聚焦「業務流程抽象」，而 Repository 層專責「資料來源抽象」。分層可避免未來資料來源變動時業務邏輯遭到污染，提升可維護性。
