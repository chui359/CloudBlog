Kubernetes Pod CPU 資源設定與 QOS 實務解析

前言

在部署微服務或雲端應用時，資源管理是確保系統穩定與高效運作的關鍵。Kubernetes（K8s）提供 CPU 與 Memory 的 request 及 limit 設定，搭配 QoS（Quality of Service）機制，協助我們分配、控制每個 Pod 的資源用量。本篇文章整理了在 EKS 架構下，如何正確理解與設定 CPU 的 limit，並說明若選擇不設 limit 會造成哪些影響，讓你能更有信心管理 K8s 資源。

介紹

1. QoS 的意思與角色

QoS（Quality of Service），原意是「服務品質」，在網路、系統領域用來描述資源分配、流量管理的策略，確保關鍵應用能夠維持最低效能標準。在 Kubernetes 中，QoS 主要用來管理 Pod 的 CPU 與 Memory 資源分配，幫助系統判斷哪些 Pod 應該優先保護或分配資源，哪些則可以在資源不足時被回收。

Kubernetes 會根據每個 Pod 的資源 request 與 limit 設定，自動分級成三種 QoS 類型：

Kubernetes 的 QoS 分級說明

  1. Guaranteed（保證等級）

    • 條件：Pod 內所有容器的 CPU 與 Memory request、limit 都有設定，而且 request 與 limit 數值完全相同。
    • 特點：資源獲得「最高保障」，最不容易被驅逐（evict），適合關鍵或需要穩定運作的服務。

QoS 的界線與管理重點

  • Guaranteed > Burstable > BestEffort
    當資源壓力大時，K8s 會優先保護 Guaranteed Pod，其次是 Burstable，而 BestEffort 最容易被驅逐。
  • 設定 request 與 limit，能讓你的服務在資源競爭時有較高優先權，降低不穩定的風險。
  • 實務建議：關鍵服務建議設成 Guaranteed，彈性服務可用 Burstable，測試或非重要服務可用 BestEffort。

2. CPU limit 與 request 的定義

  • request：Pod 啟動時向 K8s 申請的「最低保證資源」，例如 cpu: 300m 表示最少佔用 0.3 顆 CPU core。
  • limit：Pod 允許用到的「最高資源上限」，例如 cpu: 700m 表示最多用到 0.7 顆 CPU core。

3. limit 設定行為說明

  • 若 Pod 設定 limit: 700m，則該 Pod 最高只能使用 700m 的 CPU（即 0.7 core），即使系統有剩餘也不會給它更多。
  • 若未設定 limit，Pod 可依 node 可用資源「暫時暴衝」使用，但可能造成 node 內其他 Pod 競爭，影響整體效能。

4. 實務建議

  • 建議至少設定 request，讓 K8s 了解最低需求並安排合適 Node 部署。
  • limit 是否要設，視服務特性決定：
    • I/O bound 服務多半不設 limit，讓峰值負載時可彈性使用資源。
    • Critical、容易 OOM 的服務建議設 limit，避免資源被搶光導致整台 Node 不穩定。
