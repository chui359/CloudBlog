前言

在使用 FastAPI 開發專案時，常常會遇到程式碼分層不清、回傳格式不一致，或是測試不易撰寫的問題。
本篇文章將整理以下幾個常見問題，並分享如何透過 OOP 思維、抽象依賴 (interface) 與 統一回應格式 來提升程式碼的可讀性、可維護性與擴充性。

問題

  1. Service 與 API 層耦合

    • Service 直接回傳 dict 或 BaseResponse，導致難以在 CLI、排程等非 API 場景下重用。

  2. 回傳格式不一致

    • 有時回傳 dict，有時回傳 Pydantic Model，導致開發與除錯不方便。

  3. 測試困難

    • API 層直接依賴 ProjectService，無法輕易注入假實作 (Fake Service) 進行測試。

  4. 可讀性不足

    • httpx 請求、錯誤處理、資料模型轉換混雜在同一個函式，程式碼較長且難以維護。

解決辦法

1. 建立抽象介面 (Interface)

使用 ABC 定義 IProjectService，讓 API 依賴抽象而非具體實作。
這樣可以：

  • 減少耦合
  • 方便替換不同實作 (真實服務 / Mock / Cached)
  • 提升測試可行性

2. Service 專注於業務邏輯

Service 負責與外部 API 溝通，並回傳 Domain Model，而不是直接包裝成 API Response。

3. API 層包裝 BaseResponse

Router 的責任是將 Domain Model 轉成 標準回應格式 (BaseResponse)，並定義 response_model。

這樣分工清楚：

  • Service 專心處理資料
  • API 統一處理回應格式

總結

  • Interface：提升抽象層次，讓 API 依賴抽象而非實作。
  • Service：只回傳 Domain Model，專注在商業邏輯。
  • Router：負責統一回應格式，減少耦合。
  • 測試：可透過 Fake Service 注入，提升測試可維護性。

這樣的設計方式能讓程式碼 簡潔、易讀、可測試，同時符合 物件導向與乾淨架構 (Clean Architecture) 的原則。
