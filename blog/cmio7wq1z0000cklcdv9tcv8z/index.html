RabbitMQ 叢集與 Quorum Queue 高可用性實踐

前言

在分散式系統中，訊息佇列 (Message Queue) 扮演著關鍵的溝通橋樑，確保服務間的異步協作與數據可靠傳遞。RabbitMQ 作為廣受歡迎的訊息代理，其高可用性與數據一致性是生產環境中不可或缺的考量。本文旨在整理在 Kubernetes 環境下，如何正確配置 RabbitMQ 叢集所需的 Erlang Cookie，並深入理解 Quorum Queue 的目的與其在叢集中的應用，以實現更穩健的訊息處理機制。

處理問題

本次實踐主要解決以下幾個關鍵問題：

  1. RabbitMQ 叢集建立： 如何在 Kubernetes 環境中，為 RabbitMQ 節點配置必要的 Erlang Cookie，使其能夠成功形成一個穩定的叢集。
  2. Quorum Queue 應用： 理解 Quorum Queue 的設計理念，以及如何在應用程式中正確宣告並利用它來實現訊息的高可用性與強一致性。
  3. 核心概念釐清： 區分 Erlang Cookie（叢集基礎）與 Quorum Queue（佇列類型）在 RabbitMQ 生態系統中的不同職責與相互關係。

處理過程

1. 配置 Erlang Cookie 於 Kubernetes Secret

目的： Erlang Cookie 是 RabbitMQ 叢集節點間相互認證的關鍵。所有欲加入同一叢集的 RabbitMQ 節點必須擁有相同的 Erlang Cookie，否則它們將無法通信並形成叢集。在 Kubernetes 中，我們透過 Secret 來安全地管理這個敏感資訊。

處理辦法： 將 Erlang Cookie 作為環境變數 RABBITMQ_ERLANG_COOKIE 注入到 RabbitMQ Pod 中。最推薦的方式是將其加入到現有的 Kubernetes Secret 中。

2. 宣告 Quorum Queue

目的： Quorum Queue 是 RabbitMQ 提供的一種高可用性佇列類型，它利用 Raft 共識演算法來確保訊息的強一致性、高可用性和容錯性，特別適用於叢集環境。

處理辦法： 在應用程式透過 AMQP 客戶端宣告佇列時，透過 arguments 參數指定 x-queue-type 為 quorum。

  • 備註：
    • durable=True 對於 Quorum Queue 是必須的，它確保佇列定義在 RabbitMQ 重啟後依然存在。
    • arguments={"x-queue-type": "quorum"} 是關鍵，它明確指示 RabbitMQ 建立一個 Quorum Queue 而非傳統佇列。
    • Quorum Queue 透過在叢集中的多個節點間複製訊息，並使用 Raft 協定來確保訊息的一致性。當主節點故障時，會自動進行領導者選舉，實現故障轉移。

3. Erlang Cookie 與 Quorum Queue 的關係

Erlang Cookie 和 Quorum Queue 雖然都與 RabbitMQ 的高可用性相關，但它們在系統層次和功能上扮演著不同的角色，且存在明確的依賴關係：

  • Erlang Cookie：叢集的「地基」
    • 作用： 它是 RabbitMQ 節點能夠相互發現、認證並形成一個統一叢集的基礎。沒有正確配置的 Erlang Cookie，多個 RabbitMQ 實例無法組成叢集。
    • 層次： 屬於 RabbitMQ 底層的 Erlang 運行時環境配置，決定了叢集能否建立。
  • Quorum Queue：叢集上的「高可用性建築」
    • 作用： 它是 RabbitMQ 叢集建立後，在其之上提供的一種特殊佇列類型。它利用叢集的多節點特性，透過數據複製和共識演算法，實現訊息的強一致性、高可用性和自動故障轉移。
    • 層次： 屬於 RabbitMQ 提供的應用層功能，依賴於一個已經正常運作的叢集。

簡化邏輯：

可以將 Erlang Cookie 想像成是建立 RabbitMQ 叢集所需的「通行證」或「密碼」。只有持有相同通行證的節點才能進入同一個「俱樂部」（叢集）。

而 Quorum Queue 則是這個「俱樂部」裡提供的一項「VIP 服務」。這項服務（高可用性、強一致性訊息）只有在俱樂部（叢集）已經成功建立並正常運作的前提下才能提供。如果俱樂部都沒成立，VIP 服務自然無從談起。

總結： 您必須先確保 Erlang Cookie 配置正確，使 RabbitMQ 叢集能夠成功建立並穩定運行，然後才能在該叢集上有效地宣告和利用 Quorum Queue，從而為您的應用程式提供高度可靠的訊息傳遞服務。
