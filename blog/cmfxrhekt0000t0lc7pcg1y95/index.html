前言

在需要同時操作本地資料庫和外部 API 的場景中，如何確保資料一致性是個重要課題。本文探討一種簡單有效的設計方法來解決這個問題。

處理問題

傳統做法：先呼叫外部 API → 成功後更新資料庫 → 資料庫失敗時還原 API

問題點：

  1. 需要額外的 API 呼叫來還原狀態
  2. 還原操作本身也可能失敗
  3. 錯誤處理邏輯複雜

核心實作概念

1. 建立不自動 commit 的資料庫操作

2. 優化後的業務邏輯

設計優勢

  1. 簡化錯誤處理：只需要 rollback，不需要還原 API
  2. 減少網路呼叫：失敗時不需要額外的還原請求
  3. 提高一致性：資料庫和外部系統狀態更容易同步
  4. 程式碼更清晰：邏輯更直觀易懂

注意事項

使用建議：

  • 適合外部 API 回應時間較短的場景 (< 30 秒)
  • 需要注意資料庫連線佔用時間
  • 可搭配 timeout 機制避免長時間佔用資源

進階應用

適用場景：

  • 需要同時操作本地資料庫和外部 API 的業務邏輯
  • 對資料一致性有較高要求的系統
  • 希望簡化錯誤處理邏輯的專案
